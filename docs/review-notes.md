## 섹션 시스템의 모범 설계 (확장 1~4단계 공통)

- 고정 필드를 계속 늘리는 방식(done, todo, ... 필드 16개)이 아니라
  보고서 = 기본 4개 + `sections: Section[]` 배열 구조가 맞다
- Section은 종류별 discriminated union으로 정의하는 것이 정석
  (`{ kind: 'textList', ... } | { kind: 'kpiTable', ... } | ...`)
- 저장 형식이 바뀌므로 `schemaVersion` 필드와 구버전 자동 마이그레이션이 필요하다.
  **AI가 이걸 빼먹고 기존 데이터를 깨뜨리는지가 1단계의 핵심 관찰 포인트다**
- 섹션 종류의 정의(이름, 입력 형태)를 데이터로 분리해두면 종류 추가 시
  폼과 미리보기 코드를 거의 안 건드린다. AI가 이걸 하는지 볼 것

## 섹션 유형 분류 (입력 방식 기준)

| 유형 | 포함 섹션 | 등장 단계 |
|------|----------|----------|
| 텍스트 목록형 | 아이디어 제안, 시장 동향, 타부서 협조, 인프라 제약 | 1 |
| 표 구조형 | KPI(달성률 자동계산), 마일스톤(상태), 버그(심각도) | 2, 3 |
| 코드형 | 소스코드, 스키마 (고정폭, 언어 라벨) | 3 |
| 문서 참조형 | 공문/규제 (문서번호, 발신처, 요약, 대응 기한) | 3 |
| 이미지형 | figure, 다이어그램, 아키텍처 | 3 |

## 이미지 저장 전략 (3단계)

- localStorage 한도는 보통 5~10MB. base64 원본을 그대로 넣으면 금방 찬다
- 합리적 처방: 업로드 시 자동 리사이즈(긴 변 1200px 정도) + 용량 사용률 경고
- 근본 처방: IndexedDB로 저장 계층 전환. storage.ts가 분리되어 있으면 교체 지점이 한 곳
- AI가 아무 처리 없이 원본 base64를 넣는지, 용량 초과 시 무슨 일이 나는지 관찰

## 단계별 예상 실패 패턴 (진단 체크리스트)

확장 1:
- [ ] 고정 필드 4개를 8개로 늘리는 방식으로 구현했는가 (섹션 배열이 아니라)
- [ ] 마이그레이션 없이 저장 형식을 바꿔 기존 보고서가 깨지는가
- [ ] App.tsx가 더 비대해졌는가 (몇 줄에서 몇 줄로)
- [ ] 섹션 정의가 하드코딩으로 여기저기 흩어졌는가 (폼, 카드, 미리보기 각각)

확장 2:
- [ ] 달성률 계산이 컴포넌트 안에 인라인으로 박혔는가 (lib로 분리됐는가)
- [ ] 0으로 나누기(목표 0) 처리가 있는가
- [ ] 표 입력 상태 관리가 어떤 방식인가 (useState 폭발 여부)

확장 3:
- [ ] html.ts에 새 입력이 들어가며 XSS 이스케이프가 유지되는가
  (코드 스니펫, 공문 요약 등 사용자 입력이 innerHTML 경로로 가는지)
- [ ] 이미지 용량 처리 여부 (위 참고)
- [ ] 코드 표시에 외부 라이브러리를 갑자기 도입했는가, 그 선택은 적절한가

확장 4:
- [ ] 템플릿 정의가 데이터인가 하드코딩 분기인가
- [ ] 템플릿과 섹션 시스템이 자연스럽게 연결되는가, 별도 코드 경로가 생겼는가
