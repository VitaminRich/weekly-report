## 현재 위치

## 진행 리듬

확장을 연속으로 달리지 않는다. 확장과 정리를 교대한다.

```
JK 정리 0 → HY 1a → HY 1b → HY 1c → JK 정리 1 → HY 2a ... → JK 정리 2 → ...
```

이유: 확장 4단계를 연속으로 쌓으면 App.tsx god component 위에 복잡도가 곱해져서
나중에 정리할 때 확장 코드까지 다시 짜야 한다. 정리 후 확장이 쉬워지는 경험 자체가
이 프로젝트의 학습 콘텐츠다 (PROJECT.md 3.2 참고).

## 핵심 설계 결정 (모든 확장의 전제)

**고정 필드 추가가 아니라 동적 섹션 시스템으로 간다.**

- 보고서 = 기본 4개 필드(한 일, 할 일, 이슈, 메모) + 선택형 섹션 배열(`sections: Section[]`)
- 작성 화면에서 "+ 섹션 추가"로 그 주에 필요한 섹션만 골라 붙인다
- Section은 타입별 discriminated union으로 정의한다
- Report에 `schemaVersion` 필드를 추가하고, storage 로드 시 구버전 데이터를
  자동 변환하는 마이그레이션을 반드시 함께 구현한다 (CLAUDE.md 하드 규칙 3)

### 섹션 유형 분류 (입력 방식 기준)

| 유형 | 포함 섹션 | 구현 단계 |
|------|----------|----------|
| 텍스트 목록형 | 아이디어 제안, 시장 동향, 타부서 협조 및 지원 요청, 환경 셋업 및 인프라 제약 | 1b |
| 표 구조형 | KPI(지표명, 목표, 실적, 달성률 자동계산, 전주 대비), 마일스톤(항목, 목표일, 상태), 버그(제목, 심각도, 상태) | 2a, 2b, 3b |
| 코드형 | 소스코드 스니펫, 스키마 정의 (고정폭 폰트, 언어 라벨) | 3a |
| 문서 참조형 | 공문 및 규제 (문서번호, 발신처, 요약, 대응 기한) | 3b |
| 이미지형 | figure, 다이어그램, 아키텍처 그림 | 3c |

### 이미지 저장 전략 (3c에서 결정 재확인)

localStorage 한도(5~10MB) 때문에 단계적으로 간다.

1. 시작: 업로드 시 자동 리사이즈(긴 변 1200px) 후 base64로 localStorage 저장 + 용량 사용률 경고 표시
2. 용량이 문제 되면: IndexedDB로 저장 계층 전환 (storage.ts가 분리되어 있어 교체 지점은 한 곳)
3. 링크만 기록하는 방식은 미리보기에 그림이 안 떠서 채택하지 않음
