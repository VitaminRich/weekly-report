# 프로젝트 목적서 — AI 결과물을 "키울 수 있는 코드"로 만들기

> 이 문서는 앱 사용법(README.md)이 아니라, **이 저장소로 무엇을 왜 하는지**를 적은 학습·작업 기준 문서입니다.
> 앱 자체 설명은 [README.md](README.md)를 보세요.

## 1. 목표 한 줄

**AI보다 코드를 더 많이 치는 사람이 아니라, AI가 만든 결과물을 판단하고 구조화하는 사람으로 성장한다.**

Lovable / Bolt / Cursor / Claude Code / Fable 같은 도구로 비전공자가 웹앱을 만들면
겉보기 UI는 그럴듯한데 내부는 난잡한 React/Next/Vite 프로젝트가 되기 쉽다.
이 저장소는 그런 "바이브코딩 결과물"을 **프론트엔드/웹 개발자가 계속 키울 수 있는 구조로 바꾸는 과정**을
직접 실습하고 기록하기 위한 프로젝트다.

지향점: *"이 앱, 계속 키울 수 있는 구조로 바꿔드립니다"* 라고 말할 수 있는 실력.

## 2. 이 저장소가 다루는 것 (실습 대상)

**주간보고 관리 웹앱** — Fable5로 **단 하나의 프롬프트**로 생성한 코드.
React 19 + TypeScript + Vite, 서버 없이 브라우저 localStorage에 저장하는 개인용 도구.

이 코드가 학습의 "환자(patient)"다. AI가 만든 초기 결과물을 baseline으로 박제해 두고,
프롬프트 단위로 하나씩 진단 → 정리 → 구조화하며 before/after를 남긴다.

## 3. 작업 방식 (핵심 규칙)

### 3.1 1 프롬프트 = 1 커밋
- AI에게 시킨 작업(프롬프트) 하나가 커밋 하나가 된다.
- 그래야 "이 프롬프트가 만든 코드"의 문제점을 나중에 정확히 짚어낼 수 있다.
- **첫 커밋은 Fable5 원본 그대로(손대지 않은 baseline)**. 리팩토링 전/후를 비교하는 기준점.
- 프롬프트 원문은 `docs/prompts/NNN-이름.md` 로 박제하여 해당 커밋에 포함한다.

### 3.2 검증 → 정리 → 구조화 루프
개발자의 역할은 코드를 새로 짜는 게 아니라 아래를 반복하는 것:

1. **검증(Verify)** — AI 결과물이 실제로 동작하는가? 어디가 위태로운가?
2. **정리(Organize)** — 문제점을 이름 붙여 목록화한다 (예: "App.tsx가 god component").
3. **구조화(Refactor)** — 문제를 하나씩 올바른 구조로 바꾸고, 왜 그렇게 했는지 기록한다.

### 3.3 공부와 병행 + 블로그화
각 작업은 학습 주제와 연결되고, 실습 내용은 블로그 글로 정리한다.
"코드를 짜게 시키지 말고, 정리하게 시키는 법"이 최종 콘텐츠 방향.

## 5. 첫 단계 — "AI가 자주 망치는 React 패턴"

React 축부터 시작한다. AI가 흔히 잘못 짜는 패턴별로 진단하고 고친다.

| 주제 | 왜 중요한가 | 글 제목 예시 |
|------|-------------|--------------|
| useEffect 남용 | 데이터 계산, 상태 동기화, 이벤트 처리까지 useEffect에 몰아넣음 | "useEffect는 만능 도구가 아니다" |
| 상태 위치 | 전역 상태 / props drilling / local state가 뒤섞임 | "AI가 만든 상태관리 지옥 정리하기" |
| 컴포넌트 분리 | 한 파일에 500~1000줄짜리 컴포넌트가 생김 | "1000줄 React 컴포넌트 쪼개기" |
| 커스텀 훅 | API 호출, 폼 로직, UI 상태가 한 컴포넌트에 섞임 | "비즈니스 로직을 커스텀 훅으로 빼는 기준" |
| 서버/클라이언트 경계 | Next.js류에서 client component 남발 | "AI가 전부 client component로 만든 코드를 고치기" |

## 6. 이 코드베이스 초기 진단 (2026-07-07 기준)

Fable5 단일 프롬프트 결과물치고는 **의외로 정돈된 편**이다. 정직하게 기록해 둔다.

### 이미 괜찮은 것 (AI가 잘한 부분)
- 컴포넌트가 분리돼 있음: `StatsCards` / `ReportForm` / `ReportList` / `PreviewModal`
- 로직이 `lib/`로 분리됨: `date.ts` / `storage.ts` / `html.ts`
- 타입 정의 존재([src/types.ts](src/types.ts)), `ReportDraft`에 `Pick<>` 유틸리티 타입 사용
- `storage.ts`가 런타임 타입 가드로 저장 데이터 검증
- `ReportForm`에 `key={editingId ?? 'new'}` — 폼 상태 초기화를 key로 처리한 **올바른** 패턴
- 현재 useEffect 2곳 모두 **정당한 사용**: localStorage 동기화(외부 시스템), Esc 키 리스너(이벤트 구독)
- `PreviewModal`의 iframe에 `sandbox=""` 적용 — 기본적인 보안 고려 있음

### 다음 단계로 키우려면 (React 축 작업 후보)
- **App.tsx가 god component** ([src/App.tsx](src/App.tsx), 179줄): 모든 상태(reports, view, editingId, previewId, search, projectFilter, sortOrder)와 핸들러, 레이아웃이 한 곳에 몰림.
  - 데이터 로직(필터/정렬/프로젝트 목록 파생)을 `useReports` / `useReportFilters` 커스텀 훅으로 분리 후보.
  - `view` / `editingId` / `previewId`는 사실상 "현재 화면 상태" → discriminated union이나 `useReducer`로 모델링 후보.
- **ReportForm의 useState 6개** ([src/components/ReportForm.tsx](src/components/ReportForm.tsx)): 하나의 폼 상태 객체나 `useReducer`, 혹은 폼 추상화로 정리 후보.
- **파생 로직이 인라인**: 필터/정렬을 App 안 `useMemo`로 처리 — 훅으로 빼면 App은 조립만 담당.

### 다른 축에서 비어 있는 것
- 테스트 없음 (test 축) — Vitest / Testing Library / Playwright 미설정
- ESLint / Prettier 없음 (코드 품질 축)
- CI 없음 (`.github/` 없음, 배포 축)
- `html.ts`의 HTML 생성 시 XSS 이스케이프 여부 점검 필요 (보안 축)

> 위 "후보"들은 아직 손대지 않은 상태. 각 항목을 프롬프트/커밋 단위로 하나씩 처리하며 이 문서에 진행 로그를 남긴다.
